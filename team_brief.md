# TORDIT — Kickoff Brief (ไว้พูดกับทีม)

> เอกสารนี้ใช้คู่กับ spec.md/spec.docx ฉบับเต็ม. อันนี้คือสรุปสำหรับ "นัดแรก" ให้ทีมเห็นภาพ + รู้ว่าตัวเองทำอะไร แล้วเริ่มขนานกันได้เลย

---

## 1. โปรเจคนี้คืออะไร (พูด 1 นาที)

TORDIT = ระบบ **ตรวจ** เอกสาร TOR งานจัดซื้อจัดจ้างภาครัฐ ที่หน่วยงานร่างเสร็จแล้ว
- ไม่ใช่ตัวร่าง TOR ใหม่ — เป็นตัว "ตรวจของที่ร่างมาแล้ว"
- รับ PDF เข้า → บอกว่า **ผิดตรงไหน อ้างกฎข้อใด ควรแก้เป็นอะไร**
- ผิด 4 แบบ เรียงจากตรงไปตรงมา → ต้องใช้วิจารณญาณ:
  1. โครงสร้าง/ความครบ (หัวข้อครบไหม เลขข้อเพี้ยนไหม)
  2. ค่าขัดระเบียบ (ค่าปรับเกินกรอบ ระบุยี่ห้อ เกณฑ์แคบ)
  3. ความสอดคล้องภายใน (งวดงาน vs งวดเงิน วันส่งมอบนอกช่วงโครงการ)
  4. ความสมเหตุสมผล (ความเป็นมา vs วัตถุประสงค์ สัมพันธ์กันไหม)

**MVP = งานจ้างทั่วไป แบบเต็ม อย่างเดียว** แต่ทำให้ครบทุกข้อต่อตั้งแต่อัปโหลดจนออกรายงาน แล้วค่อยขยายประเภทอื่นทีหลัง

## 2. Architecture — ตรวจด้วย AI รอบเดียว (few-shot)

```
   อัปโหลด PDF (text-only, ยังไม่รองรับสแกน/OCR)
          │
          ▼
 [ผู้ใช้เลือกประเภท/แบบ]  dropdown — ผู้ใช้กำหนดเอง (ไม่ให้ LLM เดา)
          │                MVP: default = งานจ้างทั่วไป / แบบเต็ม
          ▼
 [ตรวจด้วย AI รอบเดียว]  ยิง prompt + เนื้อหา TOR → ได้ผลตรวจเลย
          │     prompt = system prompt + few-shot (ตัวอย่างผิด/ถูกจากโฟลเดอร์ 03)
          │     ไม่มี rule engine แยก ใช้ AI ล้วนเพื่อให้ทันเดโม
          ▼
 [คัดกรองการอ้างอิง]  ทิ้ง finding ที่อ้าง rule_id ไม่มีจริง (เช็คในโค้ดสั้นๆ)
          │            อ้างกฎมั่วครั้งเดียว = พังความน่าเชื่อถือทั้งระบบ
          ▼
        [รายงาน]  findings จัดกลุ่มตามหัวข้อ TOR
                  (severity + citation + suggested_fix)
```

## 3. แบ่งงาน 3 คน

| คน | บทบาท | เป็นเจ้าของ |
|----|-------|-----------|
| **ผม** | Backend / Pipeline | ดึง text PDF, ประกอบ prompt few-shot, ยิง AI, คัดกรองการอ้างอิง, API |
| **คน 2** | Frontend / UX | เว็บแอป: อัปโหลด → ยืนยันประเภท → หน้ารายงาน |
| **คน 3 (สถิติ)** | Data, Few-shot & Evaluation | คลังเอกสาร 01–03, ตัวอย่าง few-shot, ชุดทดสอบเฉลย, วัดผล |

### Backend (ผม)
- ดึง text จาก PDF (ไม่ใช้ LLM) + รับประเภท/แบบจาก dropdown
- ประกอบ prompt: system prompt + few-shot (ตัวอย่างจากคนสถิติ) + เนื้อหา TOR
- ยิง AI รอบเดียว → ได้ findings JSON (Contract) ตรงๆ
- **คัดกรองการอ้างอิง: ทิ้ง finding ที่อ้าง rule_id ไม่มีจริง เป็นโค้ด ไม่ใช่ prompt**
- API: upload → report (Contract)
- **งานแรก:** freeze Contract วันนี้ แล้ว stub API คืน mock JSON → ปลดล็อกอีก 2 คนให้เริ่มทันที

### Frontend (คน 2)
- 3 หน้า: อัปโหลด PDF → ยืนยัน/แก้ประเภท+แบบ → รายงานผลจัดกลุ่มตามหัวข้อ
- แต่ละ finding โชว์: severity (ผิดระเบียบ/ควรปรับปรุง), citation, suggested_fix
- **งานแรก:** ทำ flow ทั้ง 3 หน้าด้วย mock ตาม Contract (Findings JSON) — ไม่ต้องรอ backend เลย

### Data, Few-shot & Evaluation (คน 3 — สถิติ)
- วิเคราะห์คลัง 01–03: สรุปโครงสร้างหัวข้อจริง, ความต่างย่อ/เต็ม, pattern ข้อผิดในคู่ 03
- สร้าง **เฉลย (gold set)**: label คู่ผิด/ถูกในโฟลเดอร์ 03 (ผิดตรงไหน, rule_id อะไร, อยู่หัวข้อใด) + ฉีด error ที่รู้ชนิดลงแม่แบบสะอาดในโฟลเดอร์ 02 เพื่อเพิ่มเคส
- **คัดตัวอย่าง few-shot ให้ backend** + **แบ่ง train/test กันรั่ว**: ตัวอย่างที่ใส่ใน prompt ห้ามซ้ำกับชุดที่ใช้วัดผล ไม่งั้นคะแนนหลอกตา
- **วัดผลระบบ**: recall (จับข้อผิดที่ฝังไว้ครบไหม) + false-alarm rate (เงียบได้ไหมเมื่อ TOR ถูกอยู่แล้ว) ต่อ error class + LLM-as-judge สำหรับความสมเหตุสมผล
- **งานแรก:** อ่านคู่ 03 ทั้งหมด ทำตาราง `เอกสาร × ข้อผิด × rule_id × ตำแหน่ง` = เฉลยชุดแรกของทั้งทีม

## 4. สัญญาที่ทำให้ทำงานขนานกันได้ (freeze วันนี้)

AI รอบเดียวคืน **findings JSON** ออกมาตรงๆ ทั้งทีมตกลงหน้าตา JSON นี้ก่อนแยกย้าย แล้วทุกคนทำ mock ฝั่งตัวเองได้ทันที (frontend แสดง, สถิติวัด)

### Contract — Findings JSON (backend ผลิต, frontend แสดง, สถิติวัด)
input ที่ส่งให้ AI: `procurement_type` + `form` (ผู้ใช้เลือกจาก dropdown) + เนื้อหา TOR · output = array ของ finding หน้าตาแบบนี้
```json
{
  "error_class": 2,
  "severity": "ผิดระเบียบ",
  "topic_location": "อัตราค่าปรับ",
  "description": "ตั้งค่าปรับรายชั่วโมง ขัดรูปแบบรายวันที่ระเบียบกำหนด",
  "rule_id": "PENALTY-2",
  "citation": "ว 159 ข้อ 9 / ระเบียบ 2560",
  "suggested_fix": "เปลี่ยนเป็นค่าปรับรายวัน 0.10% ของค่าจ้าง"
}
```
หมายเหตุ: `rule_id` ต้องเป็นรหัสที่มีจริงใน checklist เท่านั้น backend เช็คในโค้ดแล้วทิ้งข้อที่อ้างกฎไม่มีจริง

## 5. ขอบเขต MVP (พูดให้ชัดว่าอะไร "ยังไม่ทำ")

**ทำ:** งานจ้างทั่วไป แบบเต็ม / PDF ที่เป็นข้อความ / ครบ flow อัปโหลด→ตรวจ→รายงาน→วัดผล
**ยังไม่ทำ:** งานซื้อ/เช่า/ก่อสร้าง/ที่ปรึกษา · แบบย่อ · วิธีเฉพาะเจาะจง · PDF สแกน/OCR · RAG (เฟสหลังค่อยเสริมบน checklist เดิม)

---

## 6. ความเสี่ยง 2 ข้อที่ต้องย้ำในที่ประชุม

1. **AI ตรวจล้วน อ่อนตรงตัวเลขเป๊ะๆ** เช่น ค่าปรับเกินกรอบ ผลงาน>50% งวดเงินรวม100 LLM พลาด threshold ได้ → ต้องวัดผลด้วยชุดเฉลย held-out (งานคน 3) ถ้าตรงนี้ไม่นิ่งค่อยเติม `if` ตรวจตัวเลขเสริม
2. **การอ้างอิงกฎคือเส้นความน่าเชื่อถือ** อ้าง rule_id ที่ไม่มีจริงแม้ครั้งเดียว = พัง → hard filter ในโค้ด (งานผม)
